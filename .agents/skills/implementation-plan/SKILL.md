## 前提

Plan 作成前に以下が揃っていること。揃っていなければユーザーに確認:

1. **対象スコープ**: 何を実装するか
2. **コードベース理解**: 既存の該当ファイル・パターンを把握済み
3. **技術スタック**: 言語・フレームワーク・テストフレームワークが判明
4. **制約条件**: 既存規約・アーキテクチャ制約が判明（`AGENTS.md` や `CLAUDE.md` 等から読み取り）

## Plan ファイルの配置先

`docs/plan/plan-<topic>.md`
(実装済みのPlanは、`docs/plan/archived/`に移動)

## Plan ファイルの必須構造（12 セクション）

以下をこの順序で含めること。セクションの省略は禁止。

### 1. ヘッダー

```markdown
# Plan: <タイトル>

<1-2文で何を実装するか>

> **Note**: 振る舞い（What）は決して変えてはいけないが、より美しい設計があれば実装方法（HOW）だけは変えてもよい。
```

Note は必ず含める。How に固執しないことを宣言する。

### 2. 設計方針

3-5 個の箇条書き。参考にした外部実装・既存パターンがあれば明記。技術的な意思決定（なぜこのアプローチか）を含める。

設計方針では、実装詳細を固定しすぎない。
特に TDD では、実装前にすべてを設計し切るのではなく、テストリスト・Red・Green・Refactor のフィードバックを通じて、必要十分な設計へ近づける。

### 3. TDD 方針

この Plan における TDD の進め方を明記する。

必ず以下を含める:

- **テストを先に大量に書かない**
  - 最初に作るのは「テストコード一式」ではなく「テストリスト / 不安リスト」
  - 各 Step では、テストリストから次に扱うテストを 1 つだけ選ぶ

- **Red / Green / Refactor を分離する**
  - Red: 期待する振る舞いを、失敗する自動テストとして表現する
  - Green: その 1 テストを通す最小限の実装をする
  - Refactor: 全テストが通る状態を保ったまま、実装コードとテストコードを整理する

- **テストは実装構造ではなく振る舞いに寄せる**
  - private メソッドや内部構造に強く依存したテストを避ける
  - 仕様・入力・出力・副作用・エラー境界を中心に確認する

- **気づきは実装に混ぜず、テストリストへ戻す**
  - Green 中に別ケースへ気づいたら、即実装せず TODO / テストリストに追加する
  - 必要ならテストの順序を組み替える

### 4. Plan スコープ

`WT作成 → 実装(TDD) → コミット(意味ごとに分離) → PR作成`

Worktree を使用しない場合は WT作成 を省略。

### 5. 対象一覧

テーブル形式で何を対象にするかを一覧化。

### 6-N. Step（TDD サイクル）

各 Step は **必ず** 以下の構成にする。

```markdown
## Step N: <モジュール名> (TDD)

### この Step の目的

<この Step で到達したい振る舞い・状態を1-2文で書く>

### テストリスト / 不安リスト

| ID | 観点 | 期待する振る舞い | 優先 |
|---|---|---|---|
| T1 | <正常系/境界/異常系など> | <期待する振る舞い> | High |
| T2 | <観点> | <期待する振る舞い> | Medium |

### 今回選ぶ 1 テスト

- 対象: `T1`
- 選ぶ理由: <最も小さく、価値があり、次の設計判断につながるため>
- この時点では扱わないこと: <T2以降、統合、最適化など>

### RED: 失敗するテストを書く

- 追加するテスト名: `test_name`
- 期待する振る舞い:
  - <Given>
  - <When>
  - <Then>
- 失敗理由の想定:
  - <未実装 / 既存仕様との不一致 / 境界未対応など>

### GREEN: 最小実装

<この 1 テストを通すための最小実装方針を書く。過剰な一般化・先回り設計は禁止。>

### REFACTOR: 設計の整理

全テストが通る状態を保ったまま、以下を確認する。

- 重複はないか
- 命名は意図を表しているか
- 責務が混ざっていないか
- テストが実装構造に依存しすぎていないか
- 次のテストに進むために十分身軽か

### テストリスト更新

- 完了: `T1`
- 追加: <Green/Refactor中に見つかった新しい不安があれば追加>
- 次候補: `T2`

### コミット

`<type>: <description>`

コミットは、RED → GREEN → REFACTOR が完了し、全テスト・型チェック・Lint が通る状態で行う。
```

**禁止事項**

- テストを最後にまとめる Step
- RED で複数のテストコードを一気に書くこと
- GREEN 中にリファクタリングを混ぜること
- Refactor を省略してコミットすること
- カバレッジ稼ぎのためだけのテストを書くこと
- 実装詳細に強く依存したテストを書くこと

テスト観点の漏れ確認は [references/test-coverage.md](references/test-coverage.md) を参照する。
ただし、各 Step ですべてのカテゴリを無理に満たす必要はない。Plan 全体として、重要な振る舞い・境界・異常系・統合観点が過不足なく扱われていることを確認する。

### 動作確認 Step

- 全テスト通過コマンド
- Lint / フォーマット / 型チェック
- 必要に応じた手動確認
- 失敗時にどこまで戻るか

### PR 作成 Step

### 変更ファイル一覧

テーブル形式。`**新規**` / `変更` を明記。

### コミット分割

Conventional Commits 形式。1 コミット = 1 論理的変更。対象ファイルを併記。

### テストリスト / テストケース一覧

- 全テストリスト項目に **連番*- を振る
- モジュールごとにグループ化
- 各項目について `完了 / 未対応 / 今回対象外` を明記
- 実際に追加する自動テスト名を併記
- セクションヘッダーに `全 N 件` を記載

### 工数見積もり

Step ごとに **行数** で見積もる。テストコードも含む。合計行数を明記。
