# Test Coverage Guide

Plan の各 Step で TDD を実践する際、テストケースは最初から網羅的に書き切らない。

このガイドは、各 Step の Red で大量のテストを書くためのものではなく、Plan 全体として重要な振る舞い・境界・異常系が漏れていないかを確認するための観点リストである。

## 基本方針

- 最初に作るのは、実装前にすべてのテストコードを書くための一覧ではなく、テストリスト / 不安リストである
- 各 Step では、テストリストから次に扱うテストを 1 つだけ選ぶ
- その 1 テストについて Red → Green → Refactor を完了してから、次のテストへ進む
- テスト観点の網羅性は、Step 単位ではなく Plan 全体で確認する
- カバレッジの数値を上げるためだけのテストは禁止する
- テストは実装構造ではなく、外から見た振る舞いに寄せる

## カテゴリ一覧

| カテゴリ       | 説明                  | 例                                                   |
| ---------- | ------------------- | --------------------------------------------------- |
| **正常系**    | 主なユースケース            | `handle_status_shows_info`, `handle_model_switch`   |
| **空・ゼロ状態** | 入力なし・データなし・初期状態     | `handle_compact_empty_session`, `list_skills_empty` |
| **境界値**    | ギリギリの値・制限・閾値        | `force_compact_runs_regardless_of_threshold`        |
| **異常系**    | エラー・失敗パス・存在しない対象    | `clear_session_idempotent_on_empty_chat`            |
| **エッジケース** | 想定外の入力・形式           | 前後空白、大文字小文字、空文字、特殊文字、null、未定義コマンド                   |
| **統合**     | モジュール間の繋ぎ込み・チャネル別動作 | `telegram_slash_does_not_enter_agent_loop`          |

## 使い方

### Step 作成時

1. この Step で実現したい振る舞いを決める
2. 関連する不安・確認観点をテストリストに書く
3. 最も小さく、次の設計判断につながる 1 テストを選ぶ
4. Red → Green → Refactor を完了する
5. 実装中に見つかった追加観点をテストリストへ戻す

### Plan 全体レビュー時

以下を確認する。

- 正常系だけに偏っていないか
- 空・ゼロ状態が考慮されているか
- 境界値が明示されているか
- 異常系が仕様として扱われているか
- エッジケースが必要以上に先回り実装されていないか
- 統合観点が最後にまとめて放置されていないか
- テストが private メソッドや内部構造に依存しすぎていないか
- Refactor の観点が各 Step に含まれているか
